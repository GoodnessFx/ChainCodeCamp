Stage 00: Prerequisites

~1 week

Before writing a single line of Solidity, you must understand the environment your code lives in. A blockchain is not a cloud server; it is a globally replicated append-only database.

Public-Key Cryptography

The foundation of identity on Ethereum. Your account is a pair of cryptographic keys:

  • Private Key: A random 256-bit number. This is your soul; if lost, your funds are gone.
  • Public Key: Derived from the private key using ECDSA.
  • Address: Derived by taking the keccak256 hash of the public key and keeping the last 20 bytes.

What is a Transaction?

A transaction is a signed message that triggers a state change. It contains:

{ "nonce": 1, // Prevent replay attacks "gasPrice": 20000000000, // Price per gas unit "gasLimit": 21000, // Max gas for this tx "to": "0x...", // Destination address "value": 1000000000000000000, // Amount in Wei (1 ETH) "data": "0x", // Input for contract calls "v, r, s": "..." // ECDSA signature components }
The Iron Rule Code is law. Bugs ARE the law. In a decentralized world, there is no "undo" button. If your contract has a vulnerability, it is a feature of the protocol until fixed.

Stage 01: EVM Internals

~2 weeks
Requires: Stage 00

The Ethereum Virtual Machine (EVM) is a stack-based state machine. Think of a stack of plates: you can only add (PUSH) or remove (POP) from the top.

The 4 Data Locations

Location Gas Cost Nature Security Significance
Stack 3-5 gas Volatile 1024 slots. Source of "Stack too deep" errors.
Memory Quadratic Volatile Expansion cost: 3n + n²/512. Layout 0x00-0x80 scratch.
Storage 20k / 2.1k Permanent Cold SLOAD (2100) and SSTORE (20000) drive all costs.
Calldata 4-16 gas Read-only Cheap input data. 4 per zero byte, 16 per non-zero.
DELEGATECALL Danger DELEGATECALL runs external code in YOUR storage context. A malicious implementation can overwrite ANY storage slot including the owner/admin slots.

Stage 02: Solidity + Yul

~4 weeks
Requires: Stage 01

Solidity is the high-level language, but Yul is the assembly language that gives you surgical control over the EVM.

Storage Layout Packing

Packing variables into a single 32-byte slot saves 20,000 gas per slot avoided.

// BAD: 3 slots (60,000 gas to initialize) uint256 a; // Slot 0 uint128 b; // Slot 1 uint256 c; // Slot 2 // GOOD: 2 slots (40,000 gas to initialize) uint256 a; // Slot 0 uint128 b; // Slot 1 (packs with d) uint128 d; // Slot 1

Slot Derivation Rules

  • Mappings: keccak256(abi.encode(key, p)) where p is the slot of the mapping.
  • Arrays: Length at slot p, elements start at keccak256(p).
Optimization Rule Reading WETH from an immutable variable costs 3 gas. Reading it from storage costs 2,100 gas. At 10,000 calls/day, this saves 20.9M gas daily.

Stage 08: Proxies & Upgrades

~2 weeks
Requires: Stage 01

Proxies allow updating code while keeping state. This is achieved via DELEGATECALL. Storage layout management is the most critical part of upgradeable contracts.

Storage Gap Warning Adding a variable in a parent contract without __gap shifts ALL child variables. Data in slot 5 moves to slot 6. State is permanently corrupted and funds are likely lost.

Uniswap V2 (x * y = k)

The Constant Product Market Maker (CPMM) formula. If you remove x, you must add y proportionally to keep k constant.

Example: Pool has 100 ETH and 200,000 USDC. Alice wants to buy 10 ETH. 90 * newUSDC = 20,000,000. New USDC = 222,222. Alice pays 22,222 USDC.

Stage 10: MEV Infrastructure

~2 weeks
Requires: Stage 01

MEV is the heartbeat of Ethereum's economy. As an architect, you must design for a world where your users' transactions are visible to bots with infinite capital.

The Sandwich Attack

A bot places a buy order before you and a sell order after you. They profit from the price slippage you cause. Protection: Always set strict minAmountOut parameters.

PBS: Proposer-Builder Separation

To prevent centralization, Ethereum separates the role of building blocks (MEV-Boost) from proposing them. This ensures smaller validators can still participate in the MEV economy.

Stage 11: Account Abstraction (ERC-4337)

~3 weeks
Requires: Stage 01

Account Abstraction removes the friction of seed phrases. It turns your wallet into a programmable smart contract.

The UserOperation

Instead of a standard transaction, users send a UserOperation to a separate mempool. A Bundler then packages these into a single L1 transaction.

Paymasters & Session Keys

  • Paymasters: Allow protocols to pay gas for their users or allow users to pay gas in stablecoins.
  • Session Keys: Grant temporary, limited permissions (e.g., "Allow this game to spend 0.1 ETH for 24 hours").

Stage 12: Zero-Knowledge

~6 weeks
Requires: Stage 00

Zero-Knowledge Proofs (ZKP) are the holy grail of privacy and scalability. You are proving computation happened correctly without revealing the data.

SNARKs vs STARKs

Feature zk-SNARK zk-STARK
Trusted Setup Required (usually) Not Required
Proof Size Very Small (~200 bytes) Larger (~hundreds of KB)
Quantum Secure No Yes

Circom Circuits

To build ZK apps, you write Circuits. These are mathematical constraints that define what is being proven.

template Multiplier() { signal input a; signal input b; signal output c; c <== a * b; // Constraint: c must be a * b }

Stage 13: Layer 2 & Bridges

~3 weeks
Requires: Stage 01

Scaling Ethereum without compromising on security. Layer 2s move execution off-chain while keeping security on-chain.

Optimistic Rollups Arbitrum, Optimism, Base. Uses Fraud Proofs. 7-day withdrawal window. Security model: "Assume valid unless proven otherwise."
ZK Rollups zkSync, Scroll, StarkNet. Uses Validity Proofs. Immediate L1 finality. Security model: "Cryptographic proof of correctness."

EIP-4844: Proto-Danksharding

The Dencun upgrade (March 2024) introduced Blobs. L2s now pay significantly less for data availability, dropping fees by 10-100x.

Sequencer Centralization Most L2s currently rely on a single sequencer. If it goes offline, transactions stop. Always check for "escape hatch" mechanisms in the protocol.

Case Study: Nomad Bridge Hack ($190M)

  1. A routine upgrade accidentally set the trusted root to 0x00.
  2. This caused any message to be considered valid by the bridge.
  3. One attacker discovered they could withdraw funds with an invalid proof.
  4. Hundreds of "copy-cat" attackers simply copied the transaction data, changed the recipient, and drained the bridge.

The veToken Model (Vote-Escrowed)

Pioneered by Curve. Lock tokens to gain voting power and boosted rewards. The longer you lock, the more power you get.

// Simplified veToken logic function createLock(uint256 amount, uint256 unlockTime) external { require(unlockTime > block.timestamp + 1 weeks); require(unlockTime <= block.timestamp + 4 years); // Power = amount * (time_remaining / 4_years) _mintPower(msg.sender, amount, unlockTime); }

Impermanent Loss (IL)
